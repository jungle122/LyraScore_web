# 数据库分析与设计实习——课程设计报告

## 实验题目

**LyraScore 云端谱仓** —— 数字乐谱资源管理与协作系统

---

## 1. 引言

### 1.1 应用内容

随着数字音乐的普及，越来越多的乐器学习者转向在线获取和管理乐谱资源。然而，传统的乐谱管理方式存在诸多痛点：本地图片散乱、练习进度无法追踪、缺少系统化的练习计划、乐谱分享渠道有限等。

LyraScore（云端谱仓）是一个面向乐器学习者的全栈协作平台，以乐谱资产管理为核心，融合系统化练习规划、练习日志记录、歌单整理、社交互动、AI 智能点歌、成就徽章激励以及实用音乐工具等功能。系统采用 B/S 架构，用户通过浏览器即可访问全部功能，所有数据实时同步到云端数据库。

本系统涵盖六大核心模块：

1. **乐谱资产管理**：支持多图上传、元数据编辑、练习状态追踪、多维筛选搜索、双页/单页阅谱模式
2. **练习计划与日志**：创建时限性练习计划并关联多首曲谱（N:M），记录每次练习时长与心得，生成聚合统计（SUM/GROUP BY）和 AI 练习总结
3. **歌单管理**：创建主题歌单，自由收藏本人和他人的公开曲谱（N:M），支持自定义排序
4. **社交与 AI 点歌**：好友添加/删除（N:M 自引用关系），通过阿里云通义千问 AI 将自由文本点歌清洗为规范的歌名+歌手
5. **成就徽章系统**：17 枚徽章，7 种触发条件，基于 Spring 事件监听器自动评估授予，支持进度条展示
6. **管理后台**：数据大盘（全站 9 项统计）、用户管理（冻结/解冻）、字典管理（快捷短语 CRUD），基于自定义 RBAC 注解

此外，系统还提供吉他定音器（5 种调弦预设 + 半音微调）、节拍器（30-250 BPM，4 种拍号）和找谱搜索（聚合 3 个乐谱网站）等实用音乐工具。

### 1.2 编写目的

本文档为 LyraScore 云端谱仓系统的数据库分析与设计实习课程设计报告，旨在：

## 2. 需求分析

### 2.1 系统功能分析

#### 2.1.1 用户角色

系统包含两类用户角色：

- **普通用户（琴友）**：注册登录后可使用全部业务功能，管理个人乐谱、练习计划、歌单、好友关系等
- **管理员**：通过后台管理面板查看全站数据、管理用户状态、维护字典配置

#### 2.1.2 功能模块总览

**模块一：乐谱资产管理**

- 上传乐谱图片（多图，最多 12 张），支持 Ctrl+V 粘贴上传
- 编辑乐谱元数据：曲名、歌手、乐器类型（吉他/尤克里里/其他）、风格（弹唱/指弹/其他）、调弦、变调夹（0-12 品）、BPM（20-300）、公开状态、乐谱备注
- 练习状态追踪：未开始 / 练习中 / 已完成
- 多维筛选搜索：按乐器、风格、练习状态筛选，关键词模糊搜索，三种排序（最新/最早/按曲名）
- 阅谱工具栏：单页/双页切换、大图预览、全屏模式
- 删除乐谱时若存在关联练习日志则禁止删除（RESTRICT 外键策略）

**模块二：练习计划与日志**

- 创建时限性练习计划（标题 + 起止日期），可关联多首曲谱（N:M）
- 记录练习日志：选择曲谱、填写时长（分钟）、达成 BPM、练习心得
- 聚合统计：累计/本周练习时长、按曲谱分组统计（GROUP BY + SUM）、近 7 天每日练习时长（DATE() + GROUP BY）
- AI 练习总结：一键生成 120-200 字的教练评语（通过通义千问 qwen-turbo），AI 失败时回退到本地模板
- 创建练习日志时自动触发徽章评估事件

**模块三：歌单管理**

- 创建/编辑/删除主题歌单（名称 + 描述）
- 从本人乐谱和他人公开乐谱中添加曲谱（N:M），唯一键防止重复
- 支持曲谱移除

**模块四：社交与 AI 点歌**

- 好友关系链：按用户名搜索添加好友、待处理请求（接受/拒绝）、好友列表（删除好友）
- AI 智能点歌：用户输入自由文本（如"周董的晴天"），后端调用通义千问清洗为规范格式（`{song, artist}`），不满意可点击重试按钮
- 点歌快捷短语：通过后台字典表配置，避免自由文本审核风险
- AI 调用失败时标记为 status=2（待人工审核），不阻塞用户请求

**模块五：成就徽章系统**

- 17 枚徽章，7 种触发条件：首次记录日志、累计练习分钟数、日志次数、曲谱数量、计划数量、歌单数量、好友数量、点歌数量
- 基于 Spring `@EventListener` 事件驱动自动评估授予
- 使用 `INSERT IGNORE` + 唯一键保证幂等性
- 未解锁徽章显示进度条（如"45 / 60 分钟"）

**模块六：管理后台**

- 数据大盘：全站 9 项统计（注册用户、上传乐谱、练习计划、练习日志、累计练习分钟、歌单、好友对、点歌请求、已颁发徽章）
- 用户管理：用户列表、冻结/解冻操作（管理员不可被冻结）
- 字典管理：快捷短语的增删改查，按 dict_type 分组
- 权限控制：基于自定义 `@RequireRole("admin")` 注解 + JWT 拦截器 + Vue 路由守卫

**附加工具**

- 吉他定音器：5 种调弦预设（Standard、Drop D、Open D、Open G、DADGAD），支持 -6 到 0 半音微调，Web Audio API 发声
- 节拍器：30-250 BPM，4 种拍号（4/4、3/4、2/4、6/8），强拍加重
- 找谱搜索：聚合 91pu、弹琴吧、不休吉他 3 个网站，简体中文自动转繁体（opencc-js）

## 3. 数据库设计

### 3.1 概念结构设计

#### 3.1.1 概念结构综述

根据需求分析，系统数据库需要存储用户、乐谱、练习计划、练习日志、歌单、好友关系、点歌请求、字典配置、徽章定义及用户徽章等数据。设计中共包含 **12 张表**，分为四个逻辑组：

- **核心实体组**：`t_user`（用户）、`t_score`（乐谱）
- **社交与互动组**：`t_friendship`（好友关系）、`t_song_request`（点歌请求）
- **练习规划组**：`t_practice_plan`（练习计划）、`t_plan_items`（计划-乐谱关联）、`t_practice_log`（练习日志）
- **歌单组织组**：`t_setlist`（歌单）、`t_setlist_items`（歌单-乐谱关联）
- **系统与游戏化组**：`t_dictionary`（字典）、`t_badge`（徽章定义）、`t_user_badge`（用户徽章）

实体间的关系涵盖三种典型模式：

| 关系类型 | 示例 | 实现方式 |
|---|---|---|
| **1:N** | 用户 → 乐谱、用户 → 练习日志 | 外键 FK 置于 N 侧 |
| **N:M** | 计划 ↔ 曲谱、歌单 ↔ 曲谱、用户 ↔ 徽章 | 关联中间表（t_plan_items / t_setlist_items / t_user_badge） |
| **自引用 N:M** | 用户 ↔ 用户（好友） | t_friendship 通过 user_id_1 和 user_id_2 分别指向 t_user |

### 3.2 逻辑结构设计

#### 3.2.1 逻辑设计综述

本系统使用 **MySQL 8.0** 关系型数据库，采用逻辑删除/状态变更（如用户封禁用 status 位）替代物理删除，外键策略依据场景区分：

- **RESTRICT**：核心业务实体（如用户-乐谱、用户-日志），防止误删引发数据孤岛
- **CASCADE**：关联中间表（计划-乐谱、歌单-乐谱），主表删除时自动清理关联记录

所有表使用 `t_` 前缀，列名采用 snake_case（下划线命名），通过 MyBatis 的 `map-underscore-to-camel-case: true` 自动映射到 Java 实体类的驼峰字段。状态标志统一使用 `tinyint` 类型，注释中注明枚举语义。

### 3.3 物理结构设计

#### 3.3.1 数据库与引擎选择

- **数据库系统**：MySQL 8.0
- **存储引擎**：InnoDB（支持事务、外键、行级锁）
- **字符集**：utf8mb4（兼容中文与 emoji）
- **排序规则**：utf8mb4_0900_ai_ci
- **数据库名**：lyrascore

#### 3.3.2 利用 DDL 描述物理数据库

以下是核心建表语句（选取 5 张代表性表，涵盖 1:N 实体表、N:M 关联表、自引用 N:M 表和字典表）：

---

**（1）t_user —— 用户表 DDL**

```sql
DROP TABLE IF EXISTS t_user;
CREATE TABLE t_user (
    id              BIGINT          NOT NULL AUTO_INCREMENT             COMMENT '用户ID',
    username        VARCHAR(50)     NOT NULL                            COMMENT '学号/用户名',
    password_hash   VARCHAR(255)    NOT NULL                            COMMENT '加密密码（BCrypt）',
    avatar_url      VARCHAR(500)    DEFAULT NULL                        COMMENT '头像相对路径',
    role            VARCHAR(20)     NOT NULL DEFAULT 'user'             COMMENT '角色：user / admin',
    status          TINYINT         NOT NULL DEFAULT 0                  COMMENT '状态：0正常 / 1封禁',
    created_at      DATETIME        NOT NULL DEFAULT CURRENT_TIMESTAMP  COMMENT '注册时间',
    PRIMARY KEY (id),
    UNIQUE KEY uk_user_username (username)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_0900_ai_ci COMMENT='用户表';
```

---

**（2）t_score —— 乐谱表 DDL（1:N 关系示例）**

```sql
DROP TABLE IF EXISTS t_score;
CREATE TABLE t_score (
    id              BIGINT          NOT NULL AUTO_INCREMENT             COMMENT '乐谱ID',
    user_id         BIGINT          NOT NULL                            COMMENT '所属用户ID',
    title           VARCHAR(100)    NOT NULL                            COMMENT '歌曲名',
    artist          VARCHAR(100)    DEFAULT NULL                        COMMENT '原唱/歌手',
    image_url       VARCHAR(2000)   NOT NULL                            COMMENT '乐谱图片相对路径，多张用英文逗号分隔',
    tuning          VARCHAR(50)     DEFAULT NULL                        COMMENT '调式（如：标准调弦 / Drop D）',
    capo            INT             DEFAULT NULL                        COMMENT '变调夹位置（0~12，NULL 表示无）',
    bpm             INT             DEFAULT NULL                        COMMENT '速度 BPM',
    is_public       TINYINT         NOT NULL DEFAULT 0                  COMMENT '是否公开：0私密 / 1公开',
    practice_status TINYINT         NOT NULL DEFAULT 0                  COMMENT '练习状态：0未开始 / 1正在练 / 2已练完',
    instrument      VARCHAR(20)     DEFAULT NULL                        COMMENT '乐器：吉他 / 尤克里里 / 其他',
    style           VARCHAR(20)     DEFAULT NULL                        COMMENT '风格：弹唱 / 指弹 / 其他',
    memo            TEXT            DEFAULT NULL                        COMMENT '备忘录/批注',
    created_at      DATETIME        NOT NULL DEFAULT CURRENT_TIMESTAMP  COMMENT '录入时间',
    PRIMARY KEY (id),
    KEY idx_score_user_created (user_id, created_at),
    CONSTRAINT fk_score_user FOREIGN KEY (user_id) REFERENCES t_user (id)
        ON DELETE RESTRICT ON UPDATE CASCADE
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_0900_ai_ci COMMENT='乐谱表';
```

> **外键策略说明**：`ON DELETE RESTRICT` 表示当 t_user 中的用户记录被尝试删除时，若该用户在 t_score 中还有乐谱，则禁止删除操作。这保证了数据完整性，防止产生孤儿记录。

---

**（3）t_friendship —— 好友关系表 DDL（自引用 N:M 关系示例）**

```sql
DROP TABLE IF EXISTS t_friendship;
CREATE TABLE t_friendship (
    id              BIGINT          NOT NULL AUTO_INCREMENT             COMMENT '关系ID',
    user_id_1       BIGINT          NOT NULL                            COMMENT '发起方用户ID',
    user_id_2       BIGINT          NOT NULL                            COMMENT '接收方用户ID',
    status          TINYINT         NOT NULL DEFAULT 0                  COMMENT '状态：0待通过 / 1已成为好友',
    created_at      DATETIME        NOT NULL DEFAULT CURRENT_TIMESTAMP  COMMENT '申请时间',
    PRIMARY KEY (id),
    UNIQUE KEY uk_friendship_pair (user_id_1, user_id_2),
    KEY idx_friendship_receiver (user_id_2, status),
    CONSTRAINT fk_friendship_user1 FOREIGN KEY (user_id_1) REFERENCES t_user (id)
        ON DELETE RESTRICT ON UPDATE CASCADE,
    CONSTRAINT fk_friendship_user2 FOREIGN KEY (user_id_2) REFERENCES t_user (id)
        ON DELETE RESTRICT ON UPDATE CASCADE
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_0900_ai_ci COMMENT='好友关系表';
```

> **设计要点**：同一张表引用 t_user 两次（user_id_1 和 user_id_2），形成自引用关系。唯一键 `uk_friendship_pair` 保证同一对用户之间最多只有一条好友记录。查询"我的好友"时使用 `CASE WHEN` 动态选取对端用户（详见 3.4.5 条件查询）。

---

**（4）t_plan_items —— 计划-乐谱关联表 DDL（N:M 关联 + CASCADE 删除）**

```sql
DROP TABLE IF EXISTS t_plan_items;
CREATE TABLE t_plan_items (
    id              BIGINT          NOT NULL AUTO_INCREMENT             COMMENT '关联主键',
    plan_id         BIGINT          NOT NULL                            COMMENT '计划ID',
    score_id        BIGINT          NOT NULL                            COMMENT '乐谱ID',
    sort_order      INT             NOT NULL DEFAULT 0                  COMMENT '排序权重（小者靠前）',
    PRIMARY KEY (id),
    UNIQUE KEY uk_plan_items_pair (plan_id, score_id),
    KEY idx_plan_items_score (score_id),
    CONSTRAINT fk_plan_items_plan  FOREIGN KEY (plan_id)  REFERENCES t_practice_plan (id)
        ON DELETE CASCADE ON UPDATE CASCADE,
    CONSTRAINT fk_plan_items_score FOREIGN KEY (score_id) REFERENCES t_score (id)
        ON DELETE CASCADE ON UPDATE CASCADE
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_0900_ai_ci COMMENT='练习计划-乐谱关联表';
```

> **外键策略说明**：此处采用 `ON DELETE CASCADE`，与 t_score 的 RESTRICT 不同。当删除某条计划或乐谱时，关联表中的记录随之自动删除，无需手动清理。CASCADE 适用于"弱的"关联关系，即关联记录脱离任一端都没有独立存在的意义。

---

**（5）t_badge —— 徽章定义表 + t_user_badge —— 用户徽章关联表 DDL**

```sql
-- 徽章定义表
DROP TABLE IF EXISTS t_badge;
CREATE TABLE t_badge (
    id              BIGINT          NOT NULL AUTO_INCREMENT             COMMENT '徽章ID',
    name            VARCHAR(50)     NOT NULL                            COMMENT '徽章名称（如：肝帝）',
    icon_url        VARCHAR(500)    DEFAULT NULL                        COMMENT '徽章图标相对路径',
    condition_type  VARCHAR(50)     NOT NULL                            COMMENT '触发条件类型（如：total_minutes / score_count）',
    condition_value INT             NOT NULL                            COMMENT '触发阈值（如：600 表示 600 分钟）',
    description     VARCHAR(255)    DEFAULT NULL                        COMMENT '获取说明',
    PRIMARY KEY (id),
    UNIQUE KEY uk_badge_name (name)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_0900_ai_ci COMMENT='成就徽章定义表';

-- 用户徽章关联表（N:M）
DROP TABLE IF EXISTS t_user_badge;
CREATE TABLE t_user_badge (
    id              BIGINT          NOT NULL AUTO_INCREMENT             COMMENT '关联主键',
    user_id         BIGINT          NOT NULL                            COMMENT '用户ID',
    badge_id        BIGINT          NOT NULL                            COMMENT '徽章ID',
    unlocked_at     DATETIME        NOT NULL DEFAULT CURRENT_TIMESTAMP  COMMENT '解锁时间',
    PRIMARY KEY (id),
    UNIQUE KEY uk_user_badge_pair (user_id, badge_id),
    KEY idx_user_badge_badge (badge_id),
    CONSTRAINT fk_user_badge_user  FOREIGN KEY (user_id)  REFERENCES t_user (id)
        ON DELETE RESTRICT ON UPDATE CASCADE,
    CONSTRAINT fk_user_badge_badge FOREIGN KEY (badge_id) REFERENCES t_badge (id)
        ON DELETE RESTRICT ON UPDATE CASCADE
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_0900_ai_ci COMMENT='用户徽章关联表';
```

> **幂等性保证**：唯一键 `uk_user_badge_pair (user_id, badge_id)` 配合 `INSERT IGNORE` 语句，确保同一用户不会重复获得同一枚徽章。无论徽章评估事件触发多少次，数据库层面保证至多插入一行。

### 3.4 数据操作

本节展示系统中典型的数据操作 SQL 语句，涵盖增、删、改、查以及条件查询和聚合统计。所有语句均来自实际的 MyBatis Mapper XML 文件。

#### 3.4.1 添加记录（INSERT）

**示例 1：注册用户**

```sql
INSERT INTO t_user (username, password_hash, role)
VALUES ('20201234001', '$2a$10$...', 'user');
```

对应 MyBatis XML：`UserMapper.xml`

```xml
<insert id="insert" parameterType="com.lyrascore.user.entity.User"
        useGeneratedKeys="true" keyProperty="id">
    INSERT INTO t_user (username, password_hash, role)
    VALUES (#{username}, #{passwordHash}, #{role})
</insert>
```

**示例 2：上传乐谱**

```sql
INSERT INTO t_score (user_id, title, artist, image_url, tuning, capo, bpm,
                     is_public, practice_status, instrument, style, memo)
VALUES (1, '晴天', '周杰伦', '/uploads/scores/uuid1.jpg', '标准调弦', 0, 120, 1, 0, '吉他', '指弹', '');
```

对应 MyBatis XML：`ScoreMapper.xml`

```xml
<insert id="insert" parameterType="com.lyrascore.score.entity.Score"
        useGeneratedKeys="true" keyProperty="id">
    INSERT INTO t_score
        (user_id, title, artist, image_url, tuning, capo, bpm, is_public,
         practice_status, instrument, style, memo)
    VALUES (#{userId}, #{title}, #{artist}, #{imageUrl}, #{tuning}, #{capo},
            #{bpm}, #{isPublic}, #{practiceStatus}, #{instrument}, #{style}, #{memo})
</insert>
```

**示例 3：徽章幂等授予（INSERT IGNORE）**

```sql
INSERT IGNORE INTO t_user_badge (user_id, badge_id)
VALUES (1, 3);
```

> `INSERT IGNORE` 在违反唯一键 `uk_user_badge_pair` 时静默跳过，不会抛出错误。这是实现"同徽章不重复颁发"的关键技术。

#### 3.4.2 删除记录（DELETE）

**示例 1：删除乐谱（仅允许删除自己的，且无关联练习日志）**

```sql
DELETE FROM t_score WHERE id = 5 AND user_id = 1;
```

对应 MyBatis XML：`ScoreMapper.xml`

```xml
<delete id="deleteByIdAndUser">
    DELETE FROM t_score WHERE id = #{id} AND user_id = #{userId}
</delete>
```

> 同时检查 `id` 和 `user_id`，防止用户越权删除他人乐谱。此外，`t_practice_log` 的外键 `fk_log_score` 采用 RESTRICT 策略，若该乐谱已有练习日志，数据库引擎会拒绝此 DELETE 操作，后端捕获异常后返回友好提示。

**示例 2：删除好友关系**

```sql
DELETE FROM t_friendship
WHERE id = 3 AND (user_id_1 = 1 OR user_id_2 = 1);
```

对应 MyBatis XML：`FriendshipMapper.xml`

```xml
<delete id="deleteByIdAndUser">
    DELETE FROM t_friendship
    WHERE id = #{id} AND (user_id_1 = #{userId} OR user_id_2 = #{userId})
</delete>
```

#### 3.4.5 条件查询（GROUP BY / JOIN / CASE WHEN / 聚合）

以下展示系统中几种较复杂的查询场景，体现了数据库课程的教学目标。

**场景 1：按曲谱分组统计练习时长（JOIN + GROUP BY + SUM）**

```sql
SELECT s.id          AS scoreId,
       s.title       AS scoreTitle,
       s.artist      AS scoreArtist,
       SUM(l.duration_mins) AS totalMins,
       COUNT(*)             AS sessions
FROM t_practice_log l
INNER JOIN t_score s ON l.score_id = s.id
WHERE l.user_id = 1
GROUP BY s.id, s.title, s.artist
ORDER BY totalMins DESC;
```

> 对应 MyBatis XML：`PracticeLogMapper.xml` → `sumByScore`。这是 GROUP BY 教学的典型案例——先 JOIN 两张表，再按曲谱分组聚合 SUM 和 COUNT。

**场景 2：按日期分组统计每日练习时长（DATE() + GROUP BY）**

```sql
SELECT DATE(log_time)         AS day,
       SUM(duration_mins)     AS totalMins
FROM t_practice_log
WHERE user_id = 1
  AND log_time >= '2026-04-01'
GROUP BY DATE(log_time)
ORDER BY day ASC;
```

> 对应 MyBatis XML：`PracticeLogMapper.xml` → `sumByDay`。`DATE()` 函数从 datetime 列中提取日期部分再分组。

**场景 3：自引用 N:M 双向好友查询（CASE WHEN 动态选对端）**

```sql
SELECT
    f.id, f.user_id_1, f.user_id_2, f.status, f.created_at,
    u.id            AS other_user_id,
    u.username      AS other_username,
    u.avatar_url    AS other_avatar
FROM t_friendship f
INNER JOIN t_user u
        ON u.id = CASE
            WHEN f.user_id_1 = 1 THEN f.user_id_2
            ELSE f.user_id_1
        END
WHERE (f.user_id_1 = 1 OR f.user_id_2 = 1)
  AND f.status = 1
ORDER BY f.created_at DESC;
```

> 对应 MyBatis XML：`FriendshipMapper.xml` → `selectFriendsOf`。由于好友关系中当前用户可能出现在 `user_id_1` 或 `user_id_2` 任一侧，使用 `CASE WHEN` 表达式动态选出"对端用户"进行 JOIN。这是自引用 N:M 关系的核心查询技术。

**场景 4：管理后台全站数据大盘（跨表 COUNT / SUM）**

```sql
SELECT
    (SELECT COUNT(*) FROM t_user)            AS totalUsers,
    (SELECT COUNT(*) FROM t_score)           AS totalScores,
    (SELECT COUNT(*) FROM t_practice_plan)   AS totalPlans,
    (SELECT COUNT(*) FROM t_practice_log)    AS totalLogs,
    (SELECT COALESCE(SUM(duration_mins), 0)
     FROM t_practice_log)                    AS totalPracticeMinutes,
    (SELECT COUNT(*) FROM t_setlist)         AS totalSetlists,
    (SELECT COUNT(*) FROM t_friendship
     WHERE status = 1)                       AS totalFriendships,
    (SELECT COUNT(*) FROM t_song_request)    AS totalSongRequests,
    (SELECT COUNT(*) FROM t_user_badge)      AS totalBadgesAwarded;
```

> 对应 MyBatis XML：`AdminMapper.xml` → `getDashboardStats`。通过一次查询中的多个子查询获取全站 9 项统计指标，避免了多次数据库往返。

**场景 5：全站累计练习分钟数**

```sql
SELECT COALESCE(SUM(duration_mins), 0)
FROM t_practice_log;
```

> `COALESCE(SUM(...), 0)` 在没有练习记录时返回 0 而非 NULL，保证前端展示友好。

**场景 6：动态查询字典表（按类型筛选且仅返回启用的条目）**

```sql
SELECT id, dict_type, dict_key, dict_value, sort_order, is_active
FROM t_dictionary
WHERE dict_type = 'quick_dm' AND is_active = 1
ORDER BY sort_order ASC;
```

> 对应 MyBatis XML：`DictionaryMapper.xml` → `selectActiveByType`。`:type` 参数化查询防止 SQL 注入。

```
│  │  GlobalExceptionHandler                      │ │
│  │  WebMvcConfig (静态资源 /uploads/**)         │ │
```

**分层职责**：

| 层 | 职责 | 示例（乐谱模块） |
|---|---|---|
| **Controller** | 接收 HTTP 请求、参数校验、调用 Service、返回 R<T> | `ScoreController`：POST /api/scores |
| **Service** | 业务逻辑、权限校验、事件发布、调用 AI | `ScoreService`：保存图片、插入记录、触发徽章评估 |

**技术选型理由**：

- **Vue 3 + Element Plus**：组件化开发效率高，Element Plus 提供成熟的表单/表格/对话框组件，适合课设快速开发 UI
- **Spring Boot 3.2**：自动配置能力强（数据源、MyBatis 扫描），约定优于配置，减少样板代码
- **MyBatis（XML Mapper）**：SQL 与 Java 分离，SQL 可独立阅读和调试。本课程的数据库教学目标是掌握 SQL 编写能力，MyBatis XML 方式保留了手写 SQL 的灵活性，而非 MyBatis-Plus 或 JPA 的自动生成
- **JWT + 自实现拦截器**：相比 Spring Security 更轻量、可理解，自定义 `@RequireRole` 注解实现声明式权限控制，代码量少且逻辑透明
- **阿里云百炼 qwen-turbo**：中文理解能力好，免费额度充足适合课设，支持 prompt 自定义（歌手艺名归一化）

### 4.6 关键技术实现

#### 4.6.1 JWT 认证与 RBAC 权限控制

本系统**不使用 Spring Security**，而是通过自定义的 `HandlerInterceptor` + JWT 实现认证和授权。选择此方案的原因是：
- 代码量约 100 行（vs Spring Security 配置数百行起步），适合理解认证全流程
- 拦截器逻辑对开发者完全透明，可单步调试
- 与 MyBatis 手工 SQL 的教学目标一以贯之——理解底层原理而非依赖框架黑盒

**核心组件**：

| 组件 | 文件 | 职责 |
|---|---|---|
| JwtUtil | `util/JwtUtil.java` | HMAC-SHA256 签名、签发/解析 token |
| JwtInterceptor | `user/interceptor/JwtInterceptor.java` | 提取 Bearer token → 解析 → 设置 UserContext → 检查 @RequireRole |
| UserContext | `user/context/UserContext.java` | ThreadLocal 存储当前请求用户信息 |
| RequireRole | `user/annotation/RequireRole.java` | 方法/类级注解，声明所需角色 |
| WebMvcConfig | `config/WebMvcConfig.java` | 注册拦截器 + 静态资源映射 |

**认证流程**：

```
请求进入 → JwtInterceptor.preHandle()
  → 提取 Authorization: Bearer <token>
  → JwtUtil.parse(token) 校验签名和过期时间
  → UserContext.set(userId, username, role)
  → 扫描方法/类上的 @RequireRole 注解
  → 如有注解且角色不匹配 → 返回 403
  → 拦截器放行 → Controller 处理
  → afterCompletion() 清除 ThreadLocal
```

> 关键代码（JwtInterceptor.java）：从 `Authorization` 头解析 JWT、从 Claims 提取用户信息、检查 `@RequireRole` 注解

```java
String header = request.getHeader("Authorization");
if (header == null || !header.startsWith("Bearer ")) {
    throw new BusinessException(401, "未登录或 token 缺失");
}
String token = header.substring("Bearer ".length()).trim();
Claims claims = jwtUtil.parse(token);
Long userId = Long.parseLong(claims.getSubject());
String role = claims.get("role", String.class);
UserContext.set(userId, username, role);

// 检查 @RequireRole 注解（方法级优先，其次类级）
RequireRole requireRole = handlerMethod.getMethodAnnotation(RequireRole.class);
if (requireRole == null) {
    requireRole = handlerMethod.getBeanType().getAnnotation(RequireRole.class);
}
if (requireRole != null && !requireRole.value().equals(role)) {
    throw new BusinessException(403, "权限不足");
}
```

#### 4.6.2 事件驱动徽章系统

传统方案（如 PRD 早期草案）建议使用**数据库触发器**，但本项目改用 Spring 的 `@EventListener` 注解。原因：

- 触发器逻辑用 SQL 编写，难以调试和测试
- 触发器隐藏在数据库层，开发者容易遗忘其存在
- Java 事件监听器可读性更好、支持断点调试、可利用 IDE 导航

**实现流程**：

```
PracticeLogService.createLog()
  → INSERT INTO t_practice_log
  → ApplicationEventPublisher.publishEvent(new PracticeLogCreatedEvent(userId))
  → PracticeLogBadgeListener.onPracticeLogCreated(event)  // @EventListener
    → BadgeService.evaluateAndAward(userId)
      → 查询所有徽章定义 + 用户当前统计
      → 遍历判断条件 → INSERT IGNORE 授予
```

> 关键代码（PracticeLogBadgeListener.java）：

```java
@Component
public class PracticeLogBadgeListener {
    @EventListener
    public void onPracticeLogCreated(PracticeLogCreatedEvent event) {
        badgeService.evaluateAndAward(event.getUserId());
    }
}
```

**幂等性保证**：`t_user_badge` 表的唯一键 `uk_user_badge_pair (user_id, badge_id)` 配合 `INSERT IGNORE`，确保同一徽章不会重复授予。

#### 4.6.3 AI 服务降级策略

AI 点歌清洗和练习总结调用阿里云百炼 qwen-turbo。系统设计了完善的降级策略：

```
AiService.cleanSongRequest(rawInput)
  → 检查 lyrascore.ai.enabled 是否开启  → 未开启返回 null
  → 检查 api-key 是否为空               → 为空返回 null
  → 调用 DashScope API  (超时/网络错误)  → 异常返回 null
  → 解析响应 JSON                       → 解析失败返回 null
  → 提取 song / artist                  → 成功返回 String[]

调用方（SongRequestService）：
  → 如果 AiService 返回 null → status = 2（待人工审核），ai_song/ai_artist 留 null
  → 用户可点击"重试"按钮重新调用 AI
```

> 核心原则：**AI 失败不阻塞用户操作**。无论 API 是否可用，用户输入的原始文本始终保存在 `raw_input` 中，后续可以人工补充或重新清洗。

#### 4.6.4 外键策略的双轨设计

本系统根据业务语义区分使用两种外键删除策略：

| 策略 | 适用场景 | 示例 |
|---|---|---|
| **RESTRICT** | 核心业务实体，子记录有独立存在意义 | `t_score → t_practice_log`：有日志的乐谱不可删除 |
| **CASCADE** | 关联中间表，脱离任一端无独立意义 | `t_practice_plan → t_plan_items`：删除计划时自动清理关联 |

这一设计体现了数据库完整性约束在实际项目中的灵活应用——不是一刀切使用 CASCADE 或 RESTRICT，而是根据业务语义逐表决策。

#### 4.6.5 自引用 N:M 关系处理

好友关系是最典型的自引用多对多关系——一条 `t_friendship` 记录连接两个 `t_user` 行。查询"我的好友"时，当前用户可能在 `user_id_1` 或 `user_id_2`：

```sql
INNER JOIN t_user u
    ON u.id = CASE
        WHEN f.user_id_1 = #{userId} THEN f.user_id_2
        ELSE f.user_id_1
    END
```

使用 `CASE WHEN` 表达式在 SQL 层面动态选择对端用户，避免了在 Java 代码中做二次判断。

```sql

CREATE DATABASE IF NOT EXISTS lyrascore
```

```bash
cd backend

# 创建表结构（12 张表）
```

```powershell
$env:DB_PASSWORD = "your_mysql_password"

```

```bash
cd backend
mvn spring-boot:run
```

看到 `Tomcat started on port(s): 8080 (http)` 即启动成功。后端监听 `http://localhost:8080/api`。

```bash
cd frontend
```

前端监听 `http://localhost:5200`，自动代理 `/api` 到后端 `:8080`。

浏览器打开 `http://localhost:5200`，进入登录页。首次使用点击"马上注册"创建账号。如需管理后台功能，需要在数据库中手动将某用户的 `role` 改为 `admin`：

```sql
UPDATE t_user SET role = 'admin' WHERE username = '你的用户名';
```

### 5.3 用户使用说明

#### 5.3.1 普通用户操作流程

1. **注册与登录**：访问 `http://localhost:5200/register`，输入用户名（建议使用学号）和密码完成注册 → 跳转登录页 → 输入用户名密码登录
2. **上传乐谱**：登录后 → 点击顶部"上传乐谱"按钮或侧边栏"谱仓" → "上传乐谱" → 选择 1-12 张图片 → 填写曲名、歌手、乐器、风格等信息 → 提交
3. **浏览与筛选乐谱**：侧边栏"谱仓"查看自己的所有乐谱 → 使用顶部筛选栏按乐器/风格/状态筛选 → 使用搜索框关键词搜索 → 点击排序下拉切换排序方式
4. **查看乐谱详情**：点击任意乐谱卡片 → 进入详情页 → 可切换单页/双页模式 → 点击图片放大查看 → 切换练习状态
5. **创建练习计划**：侧边栏"练习计划" → 点击"新建计划" → 填写标题和起止日期 → 进入计划详情 → 点击"添加曲谱"从乐谱库中多选关联曲谱
6. **记录练习日志**：在计划详情页或乐谱详情页点击"记录练习" → 选择曲谱、填写时长和心得 → 提交（系统会自动检查是否满足徽章条件）
7. **查看练习统计**：侧边栏"练习日志" → 查看累计/本周时长统计、AI 练习总结、按曲谱分组统计、近 7 天每日统计
8. **管理歌单**：侧边栏"歌单" → 新建歌单 → 进入歌单详情 → "添加曲谱"从本人或他人公开乐谱中选择
9. **添加好友**：侧边栏"社交" → "添加好友"标签 → 输入对方用户名搜索 → 发送好友请求 → 对方在"待处理"标签中接受后成为好友
10. **AI 点歌**：侧边栏"AI 点歌" → 点击"新建点歌" → 选择好友 → 输入自由文本描述想听的歌 → 提交（AI 自动清洗为规范格式） → 不满意可点击"重试"
11. **查看徽章**：侧边栏"成就徽章" → 查看已解锁和未解锁徽章 → 点击"评估徽章"手动触发评估
12. **使用音乐工具**：侧边栏"吉他定音器"调整琴弦音准、"节拍器"辅助练习节奏、"找谱搜索"聚合多个乐谱网站

#### 5.3.2 管理员操作流程

1. **登录后台**：使用 role=admin 的账号登录 → 自动跳转后台数据大盘
2. **查看数据大盘**：首页展示全站 9 项统计数据
3. **用户管理**：侧边栏"用户管理" → 查看所有注册用户 → 对违规用户点击"封禁"（管理员不可被封禁）
4. **字典管理**：侧边栏"字典管理" → 管理快捷短语的增删改查 → 修改后前端点歌页面的快捷短语列表实时更新
5. **返回用户端**：点击顶部面包屑"回到首页"切换到普通用户视角
